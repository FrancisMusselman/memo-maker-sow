What it is

Memo Maker is a working prototype built to solve a specific problem: Upland Workshop produces documents that deserve better than an email attachment.

Proposals, briefs, client-facing materials are being sent as PDFs when they could live at a URL, load cleanly on any device, and be controlled at the access level.

The application lives at memos.systems. Any team member can write a structured document with rich text and attachments, then publish it as a private mini-site with a shareable link.

Password protection, collaborator invites, drag-and-drop sections, image and PDF embeds, HTML export. I built it without a development budget using AI-assisted tools, and I'm presenting it as an app in development rather than a finished product, because that's what it is. The idea is that we take the first week to build it out together — shape it around how the team actually works before it lands in real use.

How access works

This is worth being specific about, because it matters for a studio handling confidential materials.

Every memo is private by default. Access is enforced at the database level using row-level security policies, not just hidden in the interface. A private memo is invisible to anyone who isn't the author, an invited collaborator, or an admin, whether they're accessing it through the app or directly via a URL. Unauthenticated users get a 404.

Public sharing is a deliberate choice. If you set a memo to public, anyone with the link can view it without an account. Add a password and they'll need that too. Nothing is public unless you make it public. Attached images use signed URLs that expire after one hour, so even shared assets can't be harvested after the fact.

The application has been security reviewed and is running on enforced infrastructure. For highly sensitive materials, dedicated infrastructure and a formal audit are the right next step.

Q&A
Who holds the login credentials?

Under Option 1, the application runs under Participant Systems' Lovable account. Upland Workshop holds credentials for the application itself (user logins, admin access) but not the underlying hosting account. Options 2 and 3 transfer full infrastructure ownership to Upland, removing that dependency entirely.

Where is the data actually stored?

Memos and attachments are stored in Supabase, a managed database platform. Supabase staff have infrastructure-level access as they would at any managed hosting provider. Upland's data is not accessible to other Lovable users or the public, but it does live on shared infrastructure rather than dedicated servers.

Is there any logging of who accesses a memo and when?

Not currently. There is no audit trail showing who opened a memo, when, or from where. For a studio that needs that level of accountability, it would need to be built in as part of Option 2.

After the transfer, if something breaks, who does Upland call?

After transfer under Option 3, Participant Systems is no longer in the loop. The studio would need someone comfortable working in Lovable to manage and troubleshoot the application. The handoff session is designed to make that transition real, not just theoretical.

Can Participant Systems see our memos?

Under Option 1, technically yes, as the account holder. That access would not be exercised, but it exists. Options 2 and 3 remove that access entirely.

Who reviewed the security? What does that mean specifically?

The application was reviewed against known vulnerability patterns for Supabase-backed applications. Row-level security policies were audited, storage permissions were tightened, and the public sharing logic was tested to confirm private memos are inaccessible without explicit authorization. It was not reviewed by an independent third-party security firm, which would be the appropriate next step before using this for highly sensitive materials.
